B+ Tree Overview

A balanced tree data structure optimized for fast reads, range queries, and disk-based storage systems. The foundation of most relational databases.

Structure & Characteristics

Core Properties

  • Balanced Tree: All leaf nodes at same depth, guarantees O(log n) operations
  • High Fanout: Each node has many children (100-1000), reduces tree height
  • Internal Nodes: Contain only keys for navigation (no values)
  • Leaf Nodes: Contain key-value pairs, linked as doubly-linked list
  • Sorted Order: Keys maintained in sorted order throughout tree
B+ Tree Structure (Order = 3, max 3 keys per node): [40|80] ← Root (Internal Node) / | \ / | \ [10|20] [50|60] [90|100] ← Internal Nodes / | \ / | \ / | \ / | \ / | \ / | \ [5|8] [15] [25] [45] [55] [75] [85] [95] ← Leaf Nodes (contain data) ↔ ↔ ↔ ↔ ↔ ↔ ↔ ← Linked list for range scans Internal nodes: Keys only (pointers for navigation) Leaf nodes: Key-value pairs + pointers to next/prev leaves

Node Structure

Internal Node:

  • Keys: [k1, k2, k3, ..., kn]
  • Child pointers: [p0, p1, p2, ..., pn] (n+1 pointers)
  • Property: p0 points to subtree with keys < k1, p1 points to k1 ≤ keys < k2, etc.

Leaf Node:

  • Keys: [k1, k2, k3, ..., kn]
  • Values: [v1, v2, v3, ..., vn]
  • Next pointer: Points to next leaf (for range scans)
  • Prev pointer: Points to previous leaf (for reverse scans)

Operations

Search (Point Query)

  1. Start at root, read node from disk (1 disk I/O)
  2. Binary search keys in node to find child pointer
  3. Follow pointer to child, read child node (1 disk I/O)
  4. Repeat until leaf node reached
  5. Binary search leaf for exact key
  6. Return value if found

Disk I/O: Tree height (typically 3-4 levels) = 3-4 disk reads

Time Complexity: O(log n)

Latency: 10μs - 1ms (SSD), 30-40ms (HDD with seeks)

Range Query

  1. Search for start key (same as point query)
  2. Scan forward using leaf-level linked list
  3. Read sequential leaves until end key reached
  4. Return all matching entries

Advantage: Leaves are linked, no tree traversal needed after finding start!

Performance: Very fast, sequential reads from disk

Insert

  1. Search for leaf where key should go
  2. Insert key-value in sorted position
  3. If leaf overflows (exceeds max keys):
    • Split leaf into two nodes
    • Push middle key up to parent
    • Update linked list pointers
  4. If parent overflows, recursively split upward
  5. If root splits, create new root (tree height increases)

Write Cost: 1 disk write (leaf update) + possible splits (2-3 more writes)

Issue: Random writes on HDD are slow (seek time ~10ms per write)

Delete

  1. Search for key in leaf
  2. Remove key-value pair
  3. If leaf underflows (too few keys):
    • Borrow from sibling (rebalance)
    • Or merge with sibling
  4. Update parent if needed
  5. Propagate merges upward if necessary

Space Reclaimed: Immediately (unlike LSM tombstones)

Disk Storage Details

Node = Disk Page

  • Page Size: Typically 4KB, 8KB, or 16KB
  • One node = One page: Single disk I/O reads entire node
  • Fanout Calculation: Page size / (key size + pointer size)
    • Example: 4KB page, 8-byte keys, 8-byte pointers = 4096 / 16 ≈ 256 children

Tree Height

With fanout = 256:

  • Height 1: 256 keys
  • Height 2: 256² = 65,536 keys
  • Height 3: 256³ = 16,777,216 keys
  • Height 4: 256⁴ = 4,294,967,296 keys

Result: 4 billion keys in just 4-level tree = only 4 disk reads!

Random vs Sequential I/O

B+ Tree Writes (Random):

  • Each insert updates specific leaf (random location on disk)
  • HDD: Requires seek (5-10ms) + rotation (4ms) = ~10-15ms per write
  • Throughput: ~100-200 writes/sec on HDD
  • SSD: Better but still scattered writes (~0.1-0.2ms)

B+ Tree Reads (Can be random or sequential):

  • Point queries: Random reads (tree path)
  • Range queries: Sequential reads (linked leaves)
  • Benefits from caching (hot nodes stay in RAM)

Advantages

  • Fast Point Queries: O(log n), typically 3-4 disk reads
  • Excellent Range Scans: Leaf linked list enables sequential scanning
  • Predictable Performance: Balanced structure guarantees worst-case O(log n)
  • Space Efficient: In-place updates, no old versions, 1.1-1.5x space amplification
  • Immediate Deletes: Space reclaimed instantly (no tombstones)
  • Cache Friendly: Hot nodes (upper levels) stay in page cache
  • Well Understood: Decades of research, stable implementations
  • ACID Transactions: Easier to implement strong consistency

Disadvantages

  • Slow Writes on HDD: Random disk seeks kill write throughput (100-200 writes/sec vs 50,000+ for LSM)
  • Write Amplification: Updating 1 byte requires rewriting entire 4KB page
  • Node Splits: Can cascade upward, causing multiple disk writes
  • Fragmentation: Deletes create gaps, requires periodic VACUUM/defragmentation
  • Locking Overhead: In-place updates require locks (writers block readers)
  • Poor for Write-Heavy: Bottleneck on high write throughput workloads

Use Cases

Ideal For:

  • Read-Heavy Workloads: OLTP databases (user lookups, transactions)
  • Range Queries: Reporting, analytics, time-range scans
  • Point Queries: Primary key lookups, secondary indexes
  • SSD Storage: Random I/O acceptable on SSD
  • ACID Requirements: Banking, e-commerce, inventory systems
  • Ordered Iteration: Alphabetical listings, sorted results

Examples:

  • MySQL InnoDB: Uses B+ tree for all indexes
  • PostgreSQL: B+ tree for primary and secondary indexes
  • SQLite: B+ tree variant (B-tree)
  • SQL Server: Clustered and non-clustered indexes
  • Oracle Database: Index-organized tables
  • File Systems: NTFS, ext4, XFS use B+ tree variants

Quick Comparison: B+ Tree vs LSM Tree

Aspect B+ Tree LSM Tree
Write Pattern Random (in-place) Sequential (append-only)
Write Speed (HDD) Slow (100-200/sec) Fast (50,000+/sec)
Read Speed Fast (3-4 seeks) Slower (4-20 checks)
Range Scan Excellent (linked list) Good (merge levels)
Space Usage Efficient (1.1-1.5x) Higher (1.2-3x)
Write Amplification Low (4-8x) High (10-50x)
Read Amplification Low (1x) High (4-20x)
Background Work None Heavy (compaction)
Best For Read-heavy, OLTP Write-heavy, logging

Common Optimizations

Buffer Pool / Page Cache

  • Keep hot nodes (especially upper levels) in RAM
  • Root almost always cached → saves 1 disk read
  • LRU eviction for cold pages

WAL (Write-Ahead Log)

  • Write changes to sequential log first (fast)
  • Update B+ tree pages asynchronously
  • Ensures durability without immediate random writes
  • Similar to LSM's WAL but for different purpose

Group Commit

  • Batch multiple transactions into one fsync()
  • Amortizes disk flush cost across transactions

Prefix Compression

  • Compress common key prefixes in nodes
  • Increases effective fanout (more keys per page)

Bulk Loading

  • Build tree bottom-up from sorted data
  • Much faster than individual inserts
  • Creates perfectly balanced tree

Summary

B+ Tree is a balanced, disk-optimized tree structure that excels at:

  • ✓ Fast point queries (3-4 disk reads)
  • ✓ Excellent range scans (sequential leaf traversal)
  • ✓ Predictable performance (no background compaction)
  • ✓ Space efficiency (in-place updates)

Trade-offs:

  • ✗ Slow random writes on HDD (seek time penalty)
  • ✗ Write amplification (full page updates)
  • ✗ Fragmentation over time

Best for: Read-heavy OLTP workloads, transactional databases, systems with SSD storage

Used by: MySQL, PostgreSQL, SQLite, SQL Server, Oracle, and most relational databases
