HBase Deep Dive

Why LSM Tree Alone Isn't Enough - Understanding the Need for Distributed Systems

The Question: If LSM Tree is so good, why can't I just implement it and use it as my database? Why do I need HBase?

The Answer: An LSM tree is a data structure, not a complete database system. It's like asking "I understand arrays, why do I need an operating system?" HBase solves the massive complexity of building a production-ready, distributed, fault-tolerant database system on top of LSM trees.

The Gap: LSM Tree vs Production Database

What LSM Tree Gives You (Data Structure Level)

  • ✓ Fast writes (sequential I/O)
  • ✓ MemTable (in-memory buffer)
  • ✓ SSTables (immutable on-disk files)
  • ✓ Compaction (merge files)
  • ✓ Bloom filters (skip unnecessary reads)

Scope: Single machine, single process, basic CRUD operations

What LSM Tree DOESN'T Give You

  • ✗ Distributed Storage: What if data doesn't fit on one machine?
  • ✗ High Availability: What if machine crashes?
  • ✗ Fault Tolerance: How to survive disk failures?
  • ✗ Horizontal Scalability: How to add more machines?
  • ✗ Data Replication: How to replicate data across servers?
  • ✗ Load Balancing: How to distribute requests?
  • ✗ Consistency Guarantees: How to handle concurrent writes?
  • ✗ Network Protocol: How do clients talk to server?
  • ✗ Authentication/Authorization: Security and access control
  • ✗ Monitoring/Metrics: Observability and debugging
  • ✗ Backup/Restore: Disaster recovery
  • ✗ Schema Management: Tables, columns, versioning
  • ✗ Query Language: How do users interact with it?
  • ✗ Transactions: Multi-row atomic operations
  • ✗ Secondary Indexes: Query by non-primary key

These are MASSIVE problems that take years to solve properly!

Single-Node Limitations

Let's say you implement a perfect LSM tree on a single machine. Here are the problems you'll immediately face:

1. Storage Capacity Limit

The Problem

Scenario: You're building a logging system for a large application.

  • Your application generates 10 GB of logs per day
  • You need to retain logs for 1 year = 10 GB × 365 = 3.65 TB
  • But your server only has 1 TB disk!

Your LSM tree implementation says: "Error: Disk full. Cannot write."

What do you do?

  • ❌ Buy bigger disk? (Doesn't scale indefinitely, expensive)
  • ❌ Delete old data? (Violates retention requirements)
  • ✓ Solution: Distribute data across multiple machines

2. Single Point of Failure

The Problem

Scenario: Your server crashes at 3 AM.

  • Hard drive fails
  • Or motherboard dies
  • Or power supply explodes
  • Or datacenter loses power

Result:

  • 🔴 Your entire database is DOWN
  • 🔴 All writes are REJECTED
  • 🔴 All reads are REJECTED
  • 🔴 Your application is BROKEN
  • 🔴 Users are ANGRY
  • 🔴 Money is LOST

Your LSM tree implementation says: "Sorry, I'm dead. Good luck!"

What do you do?

  • ✓ Solution: Replicate data to multiple machines
  • ✓ If one machine dies, others serve requests

3. Throughput Bottleneck

The Problem

Scenario: Your application becomes popular!

  • You go from 100 requests/sec to 100,000 requests/sec
  • Single machine CPU: 100% usage
  • Single machine network: Saturated at 10 Gbps
  • Single machine disk I/O: Maxed out at 500 MB/s

Result:

  • ⏱️ Latency increases from 1ms to 5 seconds
  • ⏱️ Timeouts everywhere
  • ⏱️ Users abandoning your app

Your LSM tree implementation says: "I'm doing my best! I can't handle more!"

What do you do?

  • ❌ Buy faster CPU? (Limited by physics, expensive)
  • ❌ Add more RAM? (Helps but doesn't solve network/disk bottleneck)
  • ✓ Solution: Distribute load across multiple machines

4. Geographic Distribution

The Problem

Scenario: You have users worldwide.

  • Your server is in New York
  • User in Tokyo makes a request
  • Round-trip latency: ~150ms (speed of light limit!)
  • User in Sydney: ~200ms

Result: Terrible user experience for non-US users

Your LSM tree implementation says: "I can't break physics!"

What do you do?

  • ✓ Solution: Deploy database servers in multiple regions
  • ✓ Route users to nearest datacenter
  • ✓ Replicate data across regions

5. Operational Complexity

Why We Need Distributed Systems

A distributed database like HBase solves all the problems above by spreading data and load across many machines.

Core Distributed System Challenges

1. Data Partitioning (Sharding)

Problem: How to split 10 TB data across 10 machines?

Simple Approach (Won't Work):

  • Put rows 1-1000 on Server 1
  • Put rows 1001-2000 on Server 2
  • Problem: What if you INSERT row 500? Which server?
  • Problem: What if Server 1 gets 90% of requests? (Hot spotting)

Better Approach (Range Partitioning):

  • Split by key ranges: A-M on Server 1, N-Z on Server 2
  • Problem: Uneven distribution (more names start with S)

HBase Approach (Regions):

  • Split table into regions (contiguous key ranges)
  • Each region: 10-100 GB of data
  • Assign regions to RegionServers (physical machines)
  • Automatic splitting when region grows too large
  • Automatic load balancing by moving regions

2. Replication

Problem: How to survive machine failures?

Solution: Keep multiple copies of each data piece

HBase Approach:

  • HBase delegates replication to HDFS (Hadoop Distributed File System)
  • Each HFile (SSTable) is replicated 3x by default
  • HDFS automatically maintains replicas
  • If one copy dies, HDFS creates new replica
Replication Example: Original Data: Region 1 (keys A-C) on RegionServer 1 HDFS Replication (3 copies): ┌─────────────────────────────────────────────────────┐ │ HFile: region1-0001.hfile (1 GB) │ ├─────────────────────────────────────────────────────┤ │ Copy 1: DataNode 1 (Primary) │ │ Copy 2: DataNode 2 (Backup) │ │ Copy 3: DataNode 3 (Backup) │ └─────────────────────────────────────────────────────┘ Failure Scenario: DataNode 1 CRASHES! 💥 HDFS Response: 1. Detects DataNode 1 is dead (heartbeat timeout) 2. Marks Copy 1 as lost 3. Automatically creates new copy on DataNode 4 4. Within minutes: 3 copies restored HBase Response: 1. RegionServer 1 loses access to local copy 2. Reads from DataNode 2 or 3 instead 3. Slight latency increase but NO DATA LOSS 4. Zero downtime!

3. Consistency and Coordination

Problem: Multiple servers, who's in charge?

Challenges:

  • Which server hosts which region?
  • What if two servers think they own the same region?
  • What if a server dies mid-write?
  • How to prevent split-brain scenarios?

HBase Solution: Apache ZooKeeper

  • Distributed coordination service
  • Stores cluster metadata
  • Elects master server
  • Tracks which regions are assigned to which servers
  • Provides distributed locks
  • Detects server failures via heartbeats

HBase Architecture

HBase is a distributed, column-oriented database built on top of HDFS, using LSM tree storage engine, coordinated by ZooKeeper.

HBase Components Deep Dive

1. HMaster (Master Server)

Responsibilities

  • Region Assignment: Decides which RegionServer hosts which region
  • Load Balancing: Moves regions to balance load
    • If RegionServer A has 100 regions and B has 10, move some from A to B
    • If RegionServer C gets 90% of requests, split hot region
  • Schema Operations: Create/delete tables, add/remove column families
  • Region Splitting: When region grows > 10 GB, split into two
  • Failure Recovery: Detects dead RegionServers, reassigns their regions
  • Monitoring: Health checks, metrics collection

High Availability

  • Active-Standby Setup: Multiple HMaster instances
  • ZooKeeper elects one as active
  • If active dies, standby takes over (automatic failover)
  • Important: HMaster is NOT on critical path for reads/writes
    • Clients talk directly to RegionServers
    • If HMaster is down, reads/writes continue!
    • Only admin operations (create table) are blocked

2. RegionServer (Worker Server)

Responsibilities

  • Serve Reads/Writes: Handle all client requests for assigned regions
  • Region Management: Host multiple regions (10-100 per server)
  • MemStore Management: In-memory write buffer (one per column family per region)
  • Flush Operations: When MemStore full, flush to HFile
  • Compaction: Minor and major compactions to merge HFiles
  • WAL: Write-ahead log for durability
  • BlockCache: LRU cache for frequently accessed blocks

Internal Structure

RegionServer Process: ┌─────────────────────────────────────────────────────┐ │ RegionServer JVM │ │ │ │ ┌─────────────────────────────────────────────┐ │ │ │ Region: users table, rows A-M │ │ │ │ │ │ │ │ ┌─────────────────────────────────────────┐ │ │ │ │ │ MemStore (RAM) │ │ │ │ │ │ - Active writes go here │ │ │ │ │ │ - Sorted by key │ │ │ │ │ │ - Size: 0-128 MB │ │ │ │ │ └─────────────────────────────────────────┘ │ │ │ │ │ │ │ │ ┌─────────────────────────────────────────┐ │ │ │ │ │ HFiles (Disk via HDFS) │ │ │ │ │ │ - Immutable SSTables │ │ │ │ │ │ - File 1: 100 MB │ │ │ │ │ │ - File 2: 150 MB │ │ │ │ │ │ - File 3: 200 MB │ │ │ │ │ └─────────────────────────────────────────┘ │ │ │ └─────────────────────────────────────────────┘ │ │ │ │ ┌─────────────────────────────────────────────┐ │ │ │ Region: users table, rows N-Z │ │ │ │ (similar structure) │ │ │ └─────────────────────────────────────────────┘ │ │ │ │ ┌─────────────────────────────────────────────┐ │ │ │ BlockCache (RAM) │ │ │ │ - LRU cache of data blocks │ │ │ │ - Size: Configurable (e.g., 40% of heap) │ │ │ │ - Shared across all regions │ │ │ └─────────────────────────────────────────────┘ │ │ │ │ ┌─────────────────────────────────────────────┐ │ │ │ WAL (Write-Ahead Log on HDFS) │ │ │ │ - Sequential log of all mutations │ │ │ │ - Shared across all regions │ │ │ │ - Rolled periodically (e.g., 1 hour) │ │ │ └─────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────┘

3. ZooKeeper

Responsibilities

  • Cluster Coordination: Central source of truth
  • Master Election: Ensures only one active HMaster
  • RegionServer Registration: All servers register with ZooKeeper
  • Root Region Location: Where is the META table?
  • Configuration: Cluster-wide settings
  • Failure Detection: Heartbeat monitoring

ZooKeeper Znodes (Data Structure)

ZooKeeper Tree: /hbase ├── /master → Current active HMaster address ├── /backup-masters → Standby HMaster list ├── /rs → RegionServer list │ ├── /rs/server1:16020 → RegionServer 1 (ephemeral node) │ ├── /rs/server2:16020 → RegionServer 2 (ephemeral node) │ └── /rs/server3:16020 → RegionServer 3 (ephemeral node) ├── /table → Table metadata ├── /table-lock → Table operation locks └── /root-region-server → Where is META region? Ephemeral Nodes: - Node exists only while server is alive - If server crashes, node disappears - Automatic failure detection!

Example: Failure Detection

Normal Operation: 1. RegionServer 2 starts 2. Creates ephemeral node: /hbase/rs/server2:16020 3. Sends heartbeat to ZooKeeper every 30 seconds 4. ZooKeeper keeps node alive Failure Scenario: 1. RegionServer 2 crashes (hardware failure) 2. No more heartbeats 3. After 60 seconds (2 missed heartbeats): - ZooKeeper deletes /hbase/rs/server2:16020 4. HMaster gets notification (watching /hbase/rs) 5. HMaster takes action: - Mark RegionServer 2 as dead - Replay WAL to recover in-flight writes - Reassign regions to other servers - Within 2-3 minutes: full recovery!

4. HDFS (Storage Layer)

Why HDFS?

  • Distributed Storage: Files split into blocks, spread across cluster
  • Replication: Each block replicated 3x (configurable)
  • Fault Tolerance: Automatic re-replication on failure
  • Scalability: Add more DataNodes to increase capacity
  • Data Locality: Computation runs where data lives (reduces network)

HFile Storage in HDFS

HBase Operations

Write Path (Put Operation)

Step-by-Step Write Flow

  1. Client Request
    • Client: put("users", "user123", "name", "Alice")
  2. Locate Region
    • Client checks local cache: Which RegionServer has "user123"?
    • If cache miss: Query META table (special system table)
    • META table returns: "user123 is in Region 5 on RegionServer 2"
    • Client caches this mapping
  3. Route to RegionServer
    • Client sends RPC to RegionServer 2
  4. RegionServer Processing
    • Step 1: WAL Write
      • Append mutation to WAL on HDFS
      • Sequential write (fast)
      • Call hsync() for durability
      • Time: ~1-5 ms
    • Step 2: MemStore Write
      • Insert into in-memory sorted structure
      • Time: ~10 microseconds
    • Step 3: Return Success
      • ACK sent to client
      • Total latency: ~1-10 ms
  5. Background: MemStore Flush
    • When MemStore reaches 128 MB (configurable)
    • Flush to new HFile on HDFS
    • Clear MemStore for new writes
    • Update META table with new HFile
Write Path Visualization: Client RegionServer 2 HDFS │ │ │ │ PUT(user123, name, Alice) │ │ ├─────────────────────────────→│ │ │ │ │ │ │ Append to WAL │ │ ├───────────────────────→│ │ │ │ │ │ WAL sync complete │ │ │←───────────────────────┤ │ │ │ │ │ Write to MemStore │ │ │ (RAM, instant) │ │ │ │ │ Success (ACK) │ │ │←─────────────────────────────┤ │ │ │ │ │ │ │ │ [Time passes, MemStore fills] │ │ │ │ │ │ Flush MemStore │ │ ├───────────────────────→│ │ │ Create HFile │ │ │ (Background async) │ │ │ │

Read Path (Get Operation)

Step-by-Step Read Flow

  1. Client Request
    • Client: get("users", "user123", "name")
  2. Locate Region
    • Same as write: Check cache or query META table
    • Route to correct RegionServer
  3. RegionServer Processing
    • Check BlockCache (RAM)
      • Is data block for "user123" cached?
      • If YES: Return immediately (1-10 microseconds)
    • Check MemStore (RAM)
      • Most recent writes are here
      • Binary search: O(log n)
      • Time: ~10 microseconds
    • Check HFiles (Disk via HDFS)
      • May need to check multiple HFiles (similar to LSM L0, L1, L2)
      • For each HFile:
        • Check Bloom filter (in cache)
        • If "maybe present": Read data block from HDFS
        • Time: ~1-10 ms per HFile
    • Merge Results
      • Combine results from MemStore + HFiles
      • Take newest version (based on timestamp)
  4. Return to Client
    • Total latency: 1-20 ms (depending on cache hits)

Region Split

Why Split?

Regions grow over time. When region exceeds threshold (e.g., 10 GB), split into two smaller regions for better load distribution.

Split Process

  1. Detection
    • RegionServer monitors region size
    • When size > 10 GB, trigger split
  2. Split Point Selection
    • Find middle key (median)
    • Example: Region has keys A-Z, split at M
  3. Create Two New Regions
    • Region 1: Keys A-M
    • Region 2: Keys N-Z
    • Important: HFiles are NOT rewritten!
      • Both regions reference same HFiles
      • Use key ranges to filter data
      • Zero data copying (instant split!)
  4. Update META Table
    • Delete old region entry
    • Add two new region entries
  5. Notify HMaster
    • HMaster may move one region to different server (load balancing)
Region Split Example: Before Split: ┌─────────────────────────────────────────────┐ │ Region 1: Keys A-Z (10 GB) │ │ RegionServer 1 │ │ │ │ HFiles: │ │ - hfile-001.hfile (2 GB) │ │ - hfile-002.hfile (3 GB) │ │ - hfile-003.hfile (5 GB) │ └─────────────────────────────────────────────┘ After Split (instant, no data copy): ┌───────────────────────────────────────────┐ │ Region 1a: Keys A-M (5 GB) │ │ RegionServer 1 │ │ │ │ HFiles (references same files): │ │ - hfile-001.hfile (read only A-M) │ │ - hfile-002.hfile (read only A-M) │ │ - hfile-003.hfile (read only A-M) │ └───────────────────────────────────────────┘ ┌───────────────────────────────────────────┐ │ Region 1b: Keys N-Z (5 GB) │ │ RegionServer 2 (moved by HMaster) │ │ │ │ HFiles (references same files): │ │ - hfile-001.hfile (read only N-Z) │ │ - hfile-002.hfile (read only N-Z) │ │ - hfile-003.hfile (read only N-Z) │ └───────────────────────────────────────────┘ Later (compaction creates region-specific HFiles): - Region 1a compacts → new HFiles with only A-M data - Region 1b compacts → new HFiles with only N-Z data - Old HFiles deleted when no longer referenced

Compaction

Minor Compaction

  • Merge small HFiles into larger ones
  • Similar to LSM tree compaction
  • Removes deleted cells (tombstones)
  • Runs automatically in background

Major Compaction

  • Merge ALL HFiles in a region into one
  • Removes all deleted cells and expired versions
  • Can be I/O intensive (read all HFiles, write new one)
  • Usually scheduled during low-traffic hours

HBase Advanced Features

1. Column Families

Concept

Group related columns together for storage and performance optimization.

Example Schema

Table: users Row Key: user123 Column Families: ├── profile: │ ├── profile:name = "Alice" │ ├── profile:age = "30" │ └── profile:email = "alice@example.com" │ └── stats: ├── stats:login_count = "1523" ├── stats:last_login = "2024-03-11" └── stats:total_posts = "42" Storage: - Each column family stored in separate HFiles - profile: Changes rarely → fewer flushes/compactions - stats: Changes frequently → more flushes Benefits: - Read only needed column families - Different compression per family - Different TTL (time-to-live) per family

Why Column Families?

  • I/O Efficiency: Read only relevant data
    • Query: "Get user profile" → Read only profile family, skip stats
    • Saves disk I/O and network bandwidth
  • Storage Optimization: Different settings per family
    • profile: GZIP compression (high ratio, slower)
    • stats: Snappy compression (fast, lower ratio)
  • Retention Policies: Different TTL
    • profile: Keep forever
    • stats: Keep only 30 days (auto-delete old versions)

2. Cell Versioning

Concept

HBase stores multiple versions of each cell, timestamped.

Example

Row Key: user123 Column: profile:name Versions (stored internally): Timestamp: 1678901234000 → "Alice" (most recent) Timestamp: 1678800000000 → "Alicia" Timestamp: 1678700000000 → "Ali" Default Behavior: - Get returns most recent: "Alice" - Can query historical: get(timestamp=1678800000000) → "Alicia" Configuration: - Max versions: 3 (keep only last 3 updates) - TTL: 30 days (auto-delete versions older than 30 days)

Use Cases

  • Audit Trail: Track all changes to sensitive data
  • Time-Series Data: Store sensor readings over time
  • Undo/History: Revert to previous state

3. Filters and Coprocessors

Filters

Server-side data filtering to reduce network transfer.

  • RowFilter: Filter rows by key pattern
  • ValueFilter: Filter by cell value
  • ColumnPrefixFilter: Get all columns starting with prefix
  • PageFilter: Limit result size (pagination)

Benefit: Process 1 million rows on server, return only 100 to client

Coprocessors

Custom code that runs on RegionServer (like database stored procedures).

Types:

  • Observer: Trigger on events (before/after put, delete, etc.)
  • Endpoint: Custom RPC endpoints for complex queries

Use Cases:

  • Real-time aggregation (count, sum, average)
  • Access control (custom authorization logic)
  • Secondary indexing

4. Bulk Loading

Problem

Loading 1 billion rows via normal PUT operations is slow (network overhead, WAL writes).

Solution: HFile Direct Upload

  1. Generate HFiles: Use MapReduce job to create sorted HFiles directly
  2. Upload to HDFS: Copy HFiles to HDFS
  3. Bulk Load: Tell HBase to "adopt" these HFiles
    • No WAL writes
    • No MemStore
    • Instant (just metadata update)

Performance: 100x faster than normal writes for large imports

5. Snapshots and Backup

Snapshots

  • Instant point-in-time copy of table
  • Uses HDFS snapshots (no data copying!)
  • Can restore or clone tables from snapshot

Backup/Restore

  • Export table to external storage (S3, another HDFS cluster)
  • Incremental backups supported
  • Cross-cluster replication for disaster recovery

Summary

Why You Need HBase (Not Just LSM Tree)

1. Distribution

  • Store petabytes across thousands of machines
  • Automatic data partitioning (regions)
  • Linear scalability (add machines = add capacity)

2. Fault Tolerance

  • 3x replication via HDFS
  • Automatic failure detection (ZooKeeper)
  • Automatic recovery (WAL replay)
  • Zero data loss on machine failures

3. High Availability

  • No single point of failure
  • Active-standby HMaster
  • Automatic failover in minutes
  • Rolling upgrades (zero downtime)

4. Operational Maturity

  • 10+ years of production use
  • Battle-tested at web scale (Yahoo, Facebook, Twitter)
  • Comprehensive monitoring and debugging tools
  • Large community and ecosystem

5. Rich Feature Set

  • Column families and versioning
  • Filters and coprocessors
  • Bulk loading and snapshots
  • Security (authentication, authorization, encryption)
Bottom Line: LSM tree is the engine, HBase is the complete car.

Just like you wouldn't build a car from scratch when you can buy a Tesla, you shouldn't build a distributed database from scratch when you can use HBase.

Focus your time on your application logic, not reinventing database infrastructure.
